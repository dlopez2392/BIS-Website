# Growth Pack: FAQ · Resource Library/Lead Magnet · Service-Area Hub — Design Spec

**Date:** 2026-07-12
**Status:** Approved (brainstorm), pending implementation plan.

## Overview

Three independent lead-gen / local-SEO features for the BIS marketing site, built on the existing stack (Next 16 App Router, next-intl v4 EN/ES, Tailwind v4 tokens, Neon + Drizzle, Resend, `StructuredData` JSON-LD, `pageMetadata`). Each is independently reviewable and deployable. Build/ship order: **FAQ → Service-Area → Resource Library/Lead Magnet** (simplest to most complex).

**Goal:** add top-of-funnel capture (lead magnet), SEO surface + objection handling (FAQ), and local-SEO signal (service-area hub) without fabricating any proof (no invented testimonials/case studies/client stories).

## Shared conventions

- New pages live under `src/app/[locale]/<route>/page.tsx`, following the existing server-component page pattern (`setRequestLocale`, `getTranslations`, `generateMetadata` via `pageMetadata`).
- All copy authored in **EN + ES with enforced parity** (a Vitest parity check per new namespace, matching the existing `missingTranslations()===[]` pattern). ES is AI-drafted then owner-reviewed (site convention: keep "IT" not "TI"; RGV-idiomatic Spanish).
- Links added to the **footer** (`footer` namespace + `Footer` component) to keep the top nav lean. The lead magnet additionally gets an inline CTA block on the home page.
- JSON-LD emitted via the existing `src/components/seo/StructuredData.tsx` component.
- Every task ends green: `npm run lint` clean, `npm run build` succeeds, relevant unit + e2e pass. ES/EN parity test green.

## Feature 1 — FAQ (`/faq`)

- **Route:** `src/app/[locale]/faq/page.tsx`. Single page.
- **Content:** ~14–16 Q&As grouped into 5 categories: **General**, **AI & Automation**, **Security & IT**, **Web & Digital**, **Working with BIS**. Rendered as grouped, expandable items (native `<details>`/`<summary>` for zero-JS accessibility, styled with tokens).
- **Data shape:** a language-neutral structure keyed by category → question ids; question/answer text lives in a new `faq` i18n namespace (EN + ES). A small `src/lib/faq.ts` (or data file) defines category → id ordering; components read i18n by id.
- **SEO:** `FAQPage` JSON-LD (question + acceptedAnswer) built from the same i18n content, injected via `StructuredData`. Page metadata via `pageMetadata`.
- **Copy:** drafted by me (EN + ES). Topics cover: what BIS does, service areas, bilingual, pricing/free-assessment framing, timelines, data handling/security (legal/medical relevance), what "bespoke" means, remote vs on-site, AI practicality, ownership/handoff, how to start.
- **Tests:** EN/ES parity for `faq` namespace; e2e renders `/en/faq` + `/es/faq` with a known heading. Optional: assert JSON-LD script present.

## Feature 2 — Service-Area Hub (`/service-area`)

- **Route:** `src/app/[locale]/service-area/page.tsx`. Single hub page (no per-city pages — avoids thin/doorway content while the site is young; expandable later).
- **Content:** "Serving the Rio Grande Valley" hero/intro, expanded city list, a short "why local matters" section, and a CTA to the free assessment / contact.
- **City list:** expand `business.areaServed` in `src/lib/seo/business.ts` to a fuller RGV set: Harlingen, McAllen, Brownsville, Edinburg, Weslaco, Mission, Pharr, San Benito, La Feria, Los Fresnos, San Juan, Alamo (final list confirmed in plan; language-neutral proper nouns). The city chips render from this single source; JSON-LD `areaServed` uses the same source.
- **SEO:** `Service` (or `ProfessionalService` `areaServed`) JSON-LD listing the cities, via `StructuredData`. Page metadata via `pageMetadata`.
- **i18n:** new `serviceArea` namespace (headings, intro, why-local copy, CTA — city names stay language-neutral).
- **Tests:** EN/ES parity for `serviceArea`; e2e renders `/en/service-area` + `/es/service-area`; unit assert every `business.areaServed` city renders.

## Feature 3 — Resource Library + Lead Magnet (`/resources`, `/resources/ai-readiness-checklist`)

### Structure
- **Extensible library.** `src/lib/resources.ts` exports a typed list: `{ slug, titleKey, blurbKey, files: { en, es } }`. Ships with one entry — **AI Readiness Checklist**. Adding the future Cybersecurity Guide = add an entry + its PDFs + i18n keys (no new plumbing).
- **`/resources`** — lists available resources (card per resource → links to its page).
- **`/resources/ai-readiness-checklist`** — resource detail page: description, what's inside, and the capture form.

### The PDF
- On-brand PDF **generated by an in-repo Node script** (`scripts/build-resources.ts` or similar) so content is versioned + regenerable; output committed to `public/resources/ai-readiness-checklist-en.pdf` and `-es.pdf`. Generation library chosen at plan time (e.g., `pdfkit`/`pdf-lib`); no runtime PDF dependency.
- Content (drafted by me): title, ~15–20 yes/no checkpoints across workflows / data / tools / team / security, a simple "where you are + first step" framing, and a "Book a free assessment" CTA. EN + ES.
- **Note:** static `public/` PDFs are directly reachable by URL (not hard-gated). Acceptable and desirable for the low-friction option-A flow.

### Gate + delivery (option A: instant + emailed copy)
- **Form:** name + email + hidden honeypot (`website`) + unchecked "Email me occasional tips" consent checkbox. Client component posting to a server action.
- **Server action `subscribeForResource`:** validates (zod), honeypot-drops, then persists + emails, mirroring the contact seam.
- **Pure orchestrator `processSubscription(input, deps)`** (deps-injected like `processContactSubmission`): `deps.insertSubscriber`, `deps.sendResourceEmail`. Never-lose-capture: ok if insert **or** email succeeds; returns `{ ok, downloadUrl }` on success.
- **On success (client):** trigger instant download of the locale-appropriate PDF (programmatic anchor click / `window.location`), show a thank-you state.
- **Confirmation email (Resend):** short, non-pitch intro + the download link again. `CONTACT_FROM` sender reused.

### Data model — new `subscribers` table (Neon/Drizzle, `src/db/schema.ts`)
| column | type | notes |
|---|---|---|
| id | uuid pk | default gen |
| email | text not null | |
| name | text | optional |
| resource | text not null | resource slug |
| locale | text not null | 'en' \| 'es' |
| newsletter_consent | boolean not null default false | |
| created_at | timestamptz default now() | |

- Separate from sales `leads` (keeps real leads clean). New migration; created in Neon (same manual-SQL approach as the `leads` table since integration conn strings are Sensitive).

### Reused / new UI
- `ResourceCTA` inline block (title + blurb + button) reused on the **home page** (below an existing section) linking to the checklist.
- Footer links: **FAQ**, **Service Area**, **Resources** (replace/augment existing footer columns).

### Tests
- `processSubscription` unit tests (ok when insert-only succeeds, email-only succeeds, both fail → error; honeypot drop). Native Vitest matchers.
- Resource-list integrity (every resource has EN+ES files + i18n keys); EN/ES parity for new `resources` namespace.
- e2e: `/resources` + checklist page render EN/ES; form present. (Live email/DB send is a manual smoke, like the contact E2E.)
